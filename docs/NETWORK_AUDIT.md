# NETWORK_AUDIT — аудит сетевых маршрутов

Дата проверки: 2026-07-19 (обновлено). **Рабочий маршрут — AmneziaWG.**

Все загрузки медиа идут **только** через настроенные outbound-профили (`OUTBOUND_REQUIRED=true`). Принцип «proxy failed ≠ try direct» соблюдён: при недоступности всех маршрутов задание завершается ошибкой `NO_ROUTE`, прямого выхода нет.

## Итог

| Маршрут | Тип | Статус | Роль |
|---|---|---|---|
| `amneziawg-main` | AmneziaWG (обфусц. WireGuard, UDP 51820) → userspace SOCKS5 | ✅ **РАБОТАЕТ** | основной |
| `xray-backup` | Xray VLESS/Reality (TCP 443) | ⚠️ троттлится upstream | резерв |
| `http-main` | HTTP proxy из config.txt | ❌ мёртв (connect timeout) | отключён |

**AmneziaWG стабилен (6/6 запросов = 200, выход 79.110.48.69, полная страница YouTube ~0.9 c.)** Через него подтверждены реальная сквозная загрузка YouTube (анализ → 480p → storage) и доставка видео в Telegram (cloud_bot, сохранён `file_id`). Обфускация Amnezia обходит троттлинг, который убивал Xray/Reality на том же сервере (`1.vpn.appswire.ru` / 79.110.48.69).

Реализация: сервис `amneziawg` (`services/amneziawg`) — `wireproxy` с поддержкой AmneziaWG, userspace gVisor-netstack, **без `/dev/net/tun` и привилегий** (целевой сервер — LXC-контейнер без модуля tun, обычный WireGuard там запустить нельзя). Конфиг передаётся через env `AWG_CONFIG` (base64), в git не хранится. Xray оставлен резервным профилем; при отказе основного маршрут-менеджер переключается на него.

## Хронология Xray (важно — узел «плавает»)

1. **Начало сессии, туннель здоров:**
   - `socks5h://xray:1080` → `api.ipify.org` = 200, exit IP `79.110.48.69`, задержка ~1.6 с (первое соединение), далее <0.5 с.
   - `youtube.com/robots.txt`, `tiktok.com/robots.txt`, `vk.com/robots.txt` → 200. `instagram.com` → 000 (típичная блокировка датацентр-диапазонов).
   - **Реальная сквозная загрузка прошла:** прямой файл `sample-5s.mp4` (2.8 МБ) скачан через Xray, проверен FFprobe, сохранён в storage, принято решение о доставке `cloud_bot`, создана подписанная ссылка. Бот получил `getMe`/`getUpdates` = 200 через тот же туннель.
2. **Через ~15 минут узел деградировал:**
   - Мелкие ответы иногда проходят, крупные — виснут; затем и мелкие перестали проходить.
   - Итоговый снимок: `api.ipify.org` через Xray = 000 (timeout), полная страница YouTube/Vimeo/archive.org = 000.
   - TCP до `1.vpn.appswire.ru:443` — **доступен** (connect 0.10 с), т.е. VLESS-сервер жив, но апстрим-выход не форвардит трафик (троттлинг/лимит бесплатного узла «NoVPN-ShareTube»).
   - Понижение MTU docker-сети до 1400 не помогло — проблема не в PMTU, а в дросселировании узла.

## Вывод

Оба предоставленных outbound-маршрута из `config.txt` на момент завершения работ **непригодны для устойчивой загрузки медиа**:
- HTTP-прокси — мёртв (connect timeout).
- Xray-узел — бесплатный, работает короткими всплесками и затем троттлится до нуля.

Это **внешнее ограничение временных доступов**, а не дефект ShareTube: система корректно маршрутизирует весь трафик загрузок через настроенный профиль, доказанно скачала реальный файл в окно работоспособности туннеля и корректно отдаёт `NO_ROUTE`, когда маршрут недоступен.

### Что требуется от владельца (одно действие)

Предоставить рабочий outbound: либо восстановить HTTP-прокси, либо дать стабильный Xray/VLESS-узел (не перегруженный бесплатный). После этого — вставить его в админке «Настройки → Сеть и прокси» (или в `XRAY_OUTBOUND_URI`/`.env`), нажать «Проверить», назначить основным. Пайплайн загрузок заработает немедленно (проверено на прямом файле).

## Как перепроверить (безопасные команды, без секретов в выводе)

```sh
# HTTP proxy (URL берётся из защищённого файла, не печатается)
curl -x "$PROXY_URL" -m 20 -s -o /dev/null -w '%{http_code}\n' https://api.ipify.org
# Xray socks inbound (внутри docker-сети)
docker run --rm --network app_st_net curlimages/curl -s -x socks5h://xray:1080 https://api.ipify.org
```

Реальные адреса, логины, пароли, UUID и subscription-URL в этом документе не публикуются.
